# Work Day - 工作日报与总结生成器

## 使用方法

| 指令 | 说明 |
|------|------|
| "生成日报" / "写日报" | 生成2条约150字的前端开发工作内容 |
| "生成总结" / "写总结" | 基于日报生成200字左右的个人思考 |

## 日报生成规则

### 核心原则
- **看起来干活了，但不知道具体干了啥**
- 内容足够专业，让人觉得在做事
- 内容足够模糊，无法追问具体细节

### 技术领域一致性
- 同一次生成的两条日报必须使用同一个技术领域
- 可选领域：Vue生态、React生态、Three.js/WebGL、构建工具链、TypeScript类型系统、Node后端等
- 两条内容围绕同一领域的不同方面展开，避免跳跃

### 标题规则
从技术框架中选取具体知识点展开，不要直接用框架名称：

| ❌ 不要这样写 | ✅ 应该这样写 |
|--------------|--------------|
| Vue响应式优化 | 基于Proxy的依赖收集机制调优 |
| React状态管理 | useReducer配合Context实现跨层级状态同步 |
| Three.js渲染优化 | 基于InstancedMesh的批量渲染策略调整 |
| Webpack配置 | SplitChunks分包策略与动态导入配合优化 |
| TypeScript类型 | 条件类型与infer关键字在泛型工具中的应用 |

### 两条日报的独立性要求
- 两条工作内容必须完全独立、毫无关联
- 属于同一技术领域，但处理不同的功能模块
- 禁止"基于上述..."、"在此基础上..."等承接表述
- 像是一天内处理了两件不相干的事

### 开头句式
- 禁止两条日报使用相同开头模式
- 每次随机生成新的开头句式，不要套用固定模板

### 内容模糊化策略
- 用词和表述每次随机变化，避免固定搭配
- 避免具体说做了什么改动、改了哪些代码

### 禁用词汇
| 类别 | 禁用词 |
|------|--------|
| 效率相关 | 效率、性能提升、优化效果、速度、耗时、显著提升、明显改善 |
| 完成相关 | 完成、完成了、搞定、做完 |
| 学习相关 | 学习、研究、探索 |
| 排查相关 | 排查、发现、定位、存在、部分、某些、个别 |
| 业务相关 | 业务、项目、需求、产品 |
| 具体数据 | 秒数、fps、百分比、MB等具体数字 |
| 语气词 | 了 |

## 总结生成规则
- 200字左右的连贯段落，不分段
- 将两条日报内容关联进行整体思考
- 包含个人感悟和技术思考
- 保持模糊风格，不具体化

## 交互流程
生成日报后提示：`是否生成总结？(y/n)`

## 注意事项
- 日报为纯文本，不包含示例代码
- 专注于"日常工作推进"的叙事角度
- 总结基于当天日报内容进行关联思考
